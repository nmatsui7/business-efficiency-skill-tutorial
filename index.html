1 What an AI-Agent Skill Is

A skill is a documented operating procedure for an AI agent. Instead of writing a one-off prompt and hoping for the best, you create a structured file that tells the agent exactly what to do, when to do it, and how to handle edge cases. The agent interprets instructions and handles natural-language ambiguity, while code handles deterministic tasks like calculations, file operations, and data validation.

Think of it this way: if you had to train a new employee every week on the same procedure, you would write it down. A skill is that written procedure, but designed for an AI agent to follow consistently.

Approach What It Is When to Use It
One-time prompt A single message you send to an AI with no saved context. The result is not reproducible and depends on how well you phrase the request each time. Quick, non-critical tasks where consistency does not matter.
Reusable template A prompt with placeholders that you fill in each time. Better than a bare prompt, but still relies on the AI understanding the intent correctly every run. Recurring tasks with a stable structure but low risk if the output varies slightly.
Python/Shell script Code that performs a specific task deterministically. Produces identical output for identical input. No AI interpretation involved. Data transformations, calculations, file operations where precision and repeatability are essential.
Agent skill A SKILL.md file combined with optional scripts. The AI reads the instructions and follows them, calling scripts for deterministic steps. Combines flexibility with reliability. Repeatable business processes that need both judgment (AI) and precision (code).
Fully integrated workflow Skills chained together with approval gates, audit logging, monitoring dashboards, and automated triggers. Enterprise-grade automation. Critical business processes with compliance requirements, high volume, and organizational oversight.

Key Insight

The power of the skill pattern is the division of labor. The AI agent handles interpretation, natural language, exceptions that require judgment, and communication with the user. Scripts handle math, file I/O, data validation, and anything that must be exactly the same every time. Neither alone is as strong as both together.

2 Start with the Business Problem

Every good skill begins with a concrete, painful business problem. Not "AI could help with reports" but "the weekly operations report takes Lisa two hours, has calculation errors in three of the last twelve weeks, and the CEO did not get it last Thursday because Lisa was out sick."

Realistic Example: Weekly Operations Report

Your team produces a weekly operations report every Monday morning. It pulls data from three spreadsheets, calculates six KPIs, formats a two-page summary, and emails it to the leadership team. The process currently takes about two hours and has several failure points.

Current Workflow

  • Manual data collection from three sources (30 min)
  • Copy-paste into a master spreadsheet (15 min)
  • Manual formula checks and corrections (20 min)
  • Format the summary document (15 min)
  • Proofread and fix inconsistencies (15 min)
  • Email to distribution list (5 min)
  • Handle "did not receive" follow-ups (20 min)

Total: ~2 hours, single point of failure

Improved Workflow with Skill

  • Skill auto-detects available data files (instant)
  • Scripts validate and combine data (1 min)
  • AI generates narrative summary (2 min)
  • Human reviews and approves (5 min)
  • Skill distributes and logs (1 min)

Total: ~9 minutes, documented and repeatable

Common Measurable Problems

  • Repetitive work: The same procedure performed weekly or more often, consuming skilled employee time on mechanical steps.
  • Inconsistent formats: Reports and outputs vary depending on who produces them, leading to confusion and rework.
  • Calculation errors: Manual formulas break when rows are added, columns shift, or data formats change.
  • Missed exceptions: Edge cases slip through because the person performing the task does not remember every rule every time.
  • Single-person dependency: Only one person knows how to do the procedure, creating risk when they are away, busy, or leave the organization.

4 Map the Existing Workflow

Before writing a skill, document the current process in detail. Every workflow has eight elements. Understanding each one ensures your skill handles the full scope.

1. Trigger

What starts the process? A scheduled time, an incoming email, a user command, or a file appearing in a folder.

2. Inputs

What data or files does the process need? Where do they come from? What formats are expected?

3. Decisions

What choices must be made during the process? Which require human judgment and which can be rule-based?

4. Transformations

What calculations, reformatting, or data manipulations happen? Are they deterministic or variable?

5. Exceptions

What can go wrong? Missing data, unexpected formats, outliers, permission failures, network errors.

6. Outputs

What is the final deliverable? A file, an email, a database update, a notification? In what format?

7. Approvals

Does someone need to review or approve the output before it is finalized or distributed?

8. Measurements

How will you know the process is working? What metrics indicate success, speed, and accuracy?

Completed Example: Weekly Operations Report

Trigger: Every Monday at 8:00 AM, or manually by the operations lead.
Inputs: sales.csv, operations.csv, support.csv from the shared drive. Each has a known column structure.
Decisions: Flag KPIs that vary more than 10% from prior week. AI decides narrative emphasis. Human approves final version.
Transformations: Sum columns, calculate rates, compute week-over-week percentage changes, format into markdown.
Exceptions: Missing file, empty CSV, column name changed, all values zero, numeric parsing failure.
Outputs: weekly_kpi_detail.csv, weekly_kpi_summary.md, email to leadership distribution list.
Approvals: Operations lead reviews summary and confirms before distribution.
Measurements: Time from trigger to delivery, error rate per report, number of manual corrections needed.

5 Anatomy of a Skill Folder

A skill is stored as a folder with a clear, standard structure. Each file has a specific purpose. The agent reads the SKILL.md file first, then uses the other files as needed.

my-skill/
├── SKILL.md              ← Main instructions the agent reads
├── scripts/              ← Deterministic helper scripts
│   ├── validate.py       ← Data validation logic
│   └── check_files.sh    ← Pre-flight checks
├── examples/             ← Sample inputs and outputs
│   ├── sample_input.csv
│   ├── sample_output.csv
│   └── sample_output.md
├── tests/                ← Automated test cases
│   ├── test_validation.py
│   └── test_edge_cases.py
└── output/               ← Generated at runtime (gitignored)
    ├── exceptions.csv
    └── summary.md

File-by-File Explanation

SKILL.md

The heart of the skill. Contains the activation description (tells the agent when to use this skill), step-by-step instructions for what the agent should do, rules about what the agent must not do, and references to any scripts or examples.

scripts/

Contains helper scripts that perform deterministic tasks. These are called by the agent during execution. Scripts handle calculations, file validation, data transformations, and anything that must be exactly the same every time.

examples/

Contains sample inputs and expected outputs. These serve two purposes: they help the user understand what the skill does, and they provide test data for verifying the skill works correctly.

tests/

Contains automated test cases that verify the skill produces correct results. Run these before deploying changes to confirm nothing is broken.

output/

A runtime directory where the skill writes its results. This folder is created during execution and should be listed in .gitignore so generated files are not committed to version control.

Progressive Disclosure

You do not need to build everything at once. Start with just the SKILL.md file. Add scripts only when you identify a step that must be deterministic. Add tests only after the skill is working. Add examples as you accumulate real data. Grow the skill as your confidence and requirements grow.

6 Writing the Skill Activation Description

The activation description is the YAML frontmatter at the top of your SKILL.md file. It is the single most important part of the skill because it determines when the agent will use it. If the description is weak, the skill will either never trigger or trigger at the wrong time.

YAML Frontmatter Example

SKILL.md frontmatter 
---
name: Weekly KPI Report Generator
description: |
  Generates the weekly KPI summary report from department CSV files.
  Use when the user asks to create, generate, build, or run the weekly
  KPI report. Triggers on phrases like "run the weekly report",
  "generate KPI summary", "build the operations report", or
  "create the Monday report". Requires sales.csv, operations.csv,
  and support.csv in the current working directory.
---

The Formula

A strong activation description answers four questions:

  1. What does it do? One sentence describing the primary output.
  2. When should it be used? Specific trigger conditions, not vague ones.
  3. What phrases might the user say? Include at least three variations.
  4. What inputs are required? List the files or data that must exist.

Weak Description

description: Helps with reports and data tasks.

This is too vague. The agent cannot distinguish this skill from dozens of others. It will either never activate or activate when it should not.

Strong Description

description: Validates expense report CSVs for missing fields, invalid dates, amounts exceeding threshold, and duplicate transactions. Use when the user asks to check, validate, audit, or review an expense report. Requires a CSV file with date, description, amount, and category columns.

Specific enough to trigger reliably. Clear enough to avoid false activation.

7 Meeting Action Register

This is a complete, no-code skill example. No Python scripts, no shell commands. The AI agent handles everything through instructions alone. This is a good starting point because it shows how far structured instructions can go.

Business Purpose

Extracts action items, decisions, and open questions from meeting notes and produces a structured action register.

Who Uses It

  • Managers who run weekly team standups and need to track follow-ups
  • Project leads who capture decisions and next steps from project meetings
  • Business analysts who summarize stakeholder meetings for distribution

Input / Output

Input

Plain-text meeting notes, either pasted directly or provided as a file path.

Output

A structured action register: table of action items, list of decisions, and list of open questions.

Why a Skill Instead of a One-Time Prompt

  • Consistent extraction rules—same criteria every time
  • No invented owners or deadlines—rules explicitly forbid it
  • Reliable output format—always structured the same way
  • Handles edge cases—vague notes, missing deadlines handled consistently

No Code Required

This skill has no scripts, no Python, and no shell commands. The entire workflow is driven by clear instructions in the SKILL.md file.

Folder Structure

Purpose of Each Folder

  • SKILL.md—Main instruction file the agent reads first.
  • references/—Domain-specific extraction rules for owners, deadlines, priorities.
  • assets/—Output template that standardizes the register format.
  • examples/—Sample inputs and expected outputs for testing.

Rendered SKILL.md

Frontmatter (YAML)

name: Meeting Action Register
A descriptive name shown in skill listings and logs.
description: Extracts action items, decisions, and owners from meeting notes.
Activation description telling the agent when to use this skill.

When to Use / When Not to Use

Use when: User asks to extract action items, create an action register, summarize a meeting, or build a follow-up list.

Do not use when: Notes are too informal, user wants a narrative summary, or task involves editing existing items.

Instructions (5 Steps)

  1. Read the meeting notes provided by the user.
  2. Identify every action item mentioned in the notes.
  3. For each action item, extract: Owner, Action, Deadline, Priority.
  4. Identify any decisions made during the meeting.
  5. List any open questions needing follow-up.

Rules (5 "Do NOT" Rules)

  • Do NOT invent owners. Write "Unassigned" if unclear.
  • Do NOT invent deadlines. Write "No deadline stated" if none mentioned.
  • Do NOT add tasks not mentioned in the notes.
  • Do NOT change the meaning of any action item.
  • If too vague, skip it and note it was unclear.

Output Format

Action Items — Table with columns: #, Owner, Action, Deadline, Priority.

Decisions — Numbered list of one-sentence summaries.

Open Questions — Numbered list of unresolved topics.

Approval Boundaries

Agent drafts the register autonomously. Human approval required before finalizing or distributing.

action-item-rules.md

Why a Separate Reference File

Rules for owner identification, deadline interpretation, and priority classification are domain knowledge that changes independently of the workflow. The agent consults this file each time it extracts an action item.

Owner Identification Rules

  • Use the exact name as written or spoken in the notes.
  • If a role is mentioned instead of a name, use the role.
  • If "someone from the team" or "we need to," write "Unassigned."

Deadline Interpretation Rules

  • Exact dates used as written.
  • Relative deadlines written as stated.
  • "ASAP" or "urgent" = High priority, "ASAP" as deadline.

Priority Classification Rules

  • High: Blocks other work, near-term deadline, or explicitly urgent.
  • Medium: Standard follow-up with defined but not urgent deadline.
  • Low: Nice-to-have, flexible or no deadlines.

8 Expense Report Validator

This skill demonstrates the division of labor between AI and code. The Python script handles deterministic validation: it reads a CSV, checks required columns, validates date formats, verifies numeric amounts, detects duplicates, and flags amounts that exceed a threshold. The AI agent interprets the results, explains them to the user in plain language, and recommends next steps.

Business Purpose

Validates expense reports for data quality issues before management review. The skill identifies invalid dates, missing fields, amounts exceeding policy thresholds, and likely duplicate transactions, then produces a structured exceptions report with recommended actions.

Division of Labor

Python Script Handles

  • CSV parsing and structural validation
  • Date format checking (YYYY-MM-DD, MM/DD/YYYY, MM-DD-YYYY)
  • Amount parsing and threshold comparison
  • Duplicate detection (same date + description + amount)
  • Empty field detection
  • Writing exceptions CSV and summary output

AI Agent Handles

  • Interpreting script results in plain language
  • Categorizing issues as confirmed errors vs. review flags
  • Explaining why each exception matters
  • Recommending corrective actions
  • Applying policy rules from the reference file
  • Presenting a management summary

Who Uses It

  • Finance teams who process bulk expense submissions and need consistent validation
  • Managers approving expenses who want a pre-screened exceptions list
  • Accountants performing audits who need structured data quality reports

Input / Output

Input

A CSV expense report with columns: date, description, amount, and category. The file can contain any number of rows.

Output

An exceptions CSV listing all rows with issues, a human-readable terminal summary, and an AI-generated interpretation of each issue with recommended actions.

Why a Skill Instead of a One-Time Prompt

  • Consistent rules—every run applies the same date formats, thresholds, and duplicate detection logic
  • Proper CSV parsing—the Python script handles encoding, quoting, and edge cases that an LLM might mishandle
  • Threshold checking—configurable dollar thresholds applied uniformly across all rows
  • Duplicate detection—deterministic matching that does not miss or hallucinate duplicates
  • Audit trail—the exceptions CSV provides a machine-readable record of every flagged issue

Folder Structure

│   └── validate_expenses.py
├── references/
│   └── expense-policy-rules.md
├── assets/
│   └── exceptions-report-template.md
└── examples/
    ├── sample-expenses.csv
    └── sample-exceptions.csv

Purpose of Each Folder

  • SKILL.md—The main instruction file. The agent reads this first to understand when to activate, what steps to follow, and what rules to enforce.
  • scripts/—Contains the Python validation script that performs deterministic checks on the CSV. Called by the agent during execution.
  • references/—Policy rules for thresholds, allowed categories, and exception handling procedures. Kept separate because these rules change independently of the validation logic.
  • assets/—The exceptions report template that standardizes how results are presented to the approver.
  • examples/—Sample input CSV and expected exceptions output. Used for demonstration, testing, and onboarding.

Why Scripts Are Essential Here

Unlike the Meeting Action Register skill, this skill requires a script because CSV parsing, numeric validation, and duplicate detection are deterministic tasks that must produce identical results every time. The AI adds value by interpreting those results, not by performing the checks themselves.

Frontmatter (YAML)

name: Expense Report Validator
A descriptive name shown in skill listings and logs.
description: Validates expense report CSVs for missing fields, invalid dates, amounts exceeding threshold, and duplicate transactions.

When to Use / When Not to Use

Use this skill when: The user provides a CSV file containing expense report data and asks to validate, check, audit, or review it for data quality issues.

Do not use this skill when:

  • Approving or rejecting expenses (the agent should never approve, reject, reimburse, or change expenses automatically)
  • Reimbursing employees
  • Calculating tax deductions
  • Creating expense reports from scratch

Instructions (10 Steps)

  1. Validate that the input is a CSV file with the expected columns: date, description, amount, category.
  2. Preserve the source file. Never modify, overwrite, or delete the original CSV.
  3. Create a separate output directory for validation results.
  4. Run the Python validator script with the appropriate threshold.
  5. Check the exit code. A non-zero exit code means exceptions were found; this is expected behavior, not an error.
  6. Inspect the generated exceptions file.
  7. Apply the policy rules in references/expense-policy-rules.md.
  8. Separate confirmed data errors (invalid dates, missing fields) from review flags (threshold exceeded, possible duplicates).
  9. Produce a management summary showing totals, exceptions, and recommended actions.
  10. Avoid approving, rejecting, reimbursing, or changing expenses automatically.

Rules

  • Never modify the source file. The original CSV must remain untouched.
  • Never approve or reject expenses. Only produce evidence and flags.
  • Never submit expenses to external systems. All actions are local.
  • Always recommend corrective actions but never take them automatically.
  • Always preserve the audit trail. The exceptions CSV is the permanent record.

Output Format

Terminal summary: File name, threshold, total rows, valid rows, exception count, and status.

Exceptions CSV: All original columns plus an issue column describing every problem found in that row.

AI interpretation: Plain-language explanation of each issue with recommended corrective actions.

Approval Boundaries

This skill produces evidence and flags only. It must not:

  • Approve expenses for reimbursement
  • Reject expenses
  • Modify expense data
  • Submit expenses to any external system

All approval and rejection decisions require explicit human action.

expense-policy-rules.md

Why a Separate Reference File

Policy thresholds, allowed categories, and exception handling procedures change independently of the validation logic. When your organization adjusts the dollar threshold or adds a new expense category, you update the reference file without touching the SKILL.md instructions or the Python script. The agent consults this file each time it interprets validation results.

Required Columns

Every expense report CSV must contain these columns:

  • date — The date the expense was incurred
  • description — A brief description of the expense
  • amount — The dollar amount (numeric)
  • category — The expense category

Accepted Date Formats

  • YYYY-MM-DD (preferred)
  • MM/DD/YYYY
  • MM-DD-YYYY

Threshold Rules

  • Default threshold: $1,000.00
  • Expenses at or above the threshold require manager approval before reimbursement.
  • Thresholds may be adjusted per department or per expense type.

Duplicate Detection Rules

A transaction is flagged as a likely duplicate if two or more rows share the same:

  • Date
  • Description
  • Amount

Common false positives: Recurring subscriptions, split reimbursements, legitimate repeated expenses (daily parking fees).

Category and Description Rules

  • "To help us resolve this faster, could you provide [specific detail]?"
  • "We have escalated this to a specialist who will contact you within [timeframe]."
  • "Thank you for bringing this to our attention."

Prohibited Phrases

  • "This is not our fault" — Never assign blame to the customer.
  • "You should have..." — Never imply the customer made a mistake.
  • "I don't know" — Use "Let me connect you with someone who can help."
  • "That's not my department" — Use "I will route this to the right team."
  • "Calm down" — Never tell a customer how to feel.

How the Skill Executes

The agent follows these ten steps. Steps 1–2 are trigger and input. Steps 3–5 are classification. Step 6 is the confidence gate. Steps 7–8 are response drafting. Steps 9–10 are the approval gate.

User provides request → Agent reads request → Agent applies routing rules → Agent assigns priority → Agent assesses confidence → Agent flags low confidence → Agent composes response → Agent presents decision → Human reviews → Human confirms or overrides

Step-by-Step

  1. User provides a customer request (pasted directly or as a file path).
  2. Agent reads the request in full, identifying customer name, core issue, product references, and urgency signals.
  3. Agent consults references/routing-rules.md to identify the target team.
  4. Agent consults references/priority-definitions.md to assign a priority level.
  5. Agent assesses confidence (0–100%) based on clarity of routing and priority signals.
  6. If confidence < 80%, agent flags the request for human review.
  7. Agent consults references/approved-response-language.md for response phrasing.
  8. Agent composes a suggested response using only approved language.
  9. Agent presents the complete triage decision: team, priority, confidence, suggested response.
  10. Human reviews and confirms or overrides the decision, then approves or edits the response.

11 Choosing Between Tools

No single tool is best for everything. The right choice depends on what the task requires. Most real skills combine multiple tools.

AI Agent

  • Interpreting natural language input
  • Generating narrative text and summaries
  • Making judgment calls on ambiguous data
  • Communicating with users in conversational language
  • Classifying unstructured text
  • Drafting emails, reports, and responses

Python Script

  • CSV, JSON, or XML parsing and validation
  • Numerical calculations and statistics
  • Data transformation and formatting
  • API calls with authentication
  • Date and currency manipulation
  • Duplicate detection and deduplication

Shell Script

  • File existence and permission checks
  • Directory creation and management
  • Environment validation (is Python installed?)
  • Running other scripts in sequence
  • Simple string matching and logging
  • Setting up pre-flight checks

Spreadsheets

  • Quick one-off data exploration
  • Sharing results with non-technical stakeholders
  • Simple pivot tables and charts
  • Prototyping formulas before coding them
  • Data entry by multiple team members
  • When the audience prefers a familiar format

Combine Tools

The most effective skills use multiple tools together. For example: a shell script verifies that input files exist, a Python script validates and transforms the data, and the AI agent interprets the results and communicates them to the user. Each tool does what it does best.

When your skill includes scripts that run automatically, safety is paramount. A script that silently overwrites a file or leaks credentials can do more damage than the manual process it replaced. Use this checklist to ensure every script meets minimum safety standards.

13 Business Controls and Governance

Automation without governance is speed without steering. As you deploy skills that affect real business data and customer interactions, you need controls to ensure correctness, compliance, and accountability.

Data Privacy

Understand what data the skill touches. Personal data, financial data, and health data each have different regulatory requirements. Ensure the skill complies with relevant regulations (GDPR, HIPAA, SOC 2).

Confidential Information

Skills should not expose confidential business information in logs, error messages, or output files that could be seen by unauthorized parties.

Personally Identifiable Information (PII)

If the skill processes PII, ensure it is handled according to your organization's data handling policies. Minimize the PII that appears in logs and intermediate files.

Access Controls

Limit who can run, modify, and approve skills. Not everyone should have write access to production skills.

Approval Thresholds

Define clear thresholds for when human approval is required. For example, any expense over a dollar amount or any customer-facing communication must be approved.

Segregation of Duties

The person who writes the skill should not be the only person who can approve its use in production. Separate development from approval.

Audit Trails

Log who ran the skill, when, what inputs were used, what decisions were made, and what outputs were produced. These logs are essential for troubleshooting and compliance.

Record Retention

Define how long inputs, outputs, and logs are retained. Automate cleanup to prevent unbounded storage growth.

Version Control

Store all skill files in a version control system. Every change should be committed with a message explaining what changed and why.

Change Management

Follow your organization's change management process for skills that affect production systems. Document changes, get approvals, and schedule deployments.

Script Review

Review scripts before they are added to a skill. Check for safety issues, performance problems, and compliance with the safe scripting standards.

Skill Ownership

Every skill should have a designated owner responsible for maintenance, updates, and retirement. Unowned skills become technical debt.

Warning

An efficient incorrect process can create larger problems faster. Speed without accuracy multiplies the impact of errors. Always validate that a skill produces correct results before removing human oversight.

14 Testing a Skill

Testing ensures that a skill produces the expected results under various conditions. A skill that has not been tested is a skill that might fail silently in production.

Test Folder Structure

tests/
├── test_validation.py       ← Unit tests for validation logic
├── test_edge_cases.py       ← Boundary and error conditions
├── test_integration.py      ← End-to-end skill execution
├── fixtures/                ← Test data files
│   ├── valid_report.csv
│   ├── missing_columns.csv
│   ├── empty_file.csv
│   └── duplicate_rows.csv
└── expected/                ← Expected outputs for comparison
    ├── valid_exceptions.csv
    └── valid_summary.md

Five Testing Categories

1. Unit Tests

Test individual functions in isolation. Verify that parse_date handles all formats, that detect_duplicates catches duplicates, and that parse_amount rejects non-numeric input.

2. Edge Case Tests

Test boundary conditions: empty files, files with one row, files with thousands of rows, maximum and minimum numeric values, special characters in text fields.

3. Integration Tests

Test the full skill from trigger to output. Provide sample input, run the skill, and verify that all outputs match expected results.

4. Regression Tests

When a bug is found and fixed, write a test that reproduces the bug. This ensures the same problem never returns.

5. Acceptance Tests

Verify the skill from the user's perspective. Does the output meet the business requirements? Is the format correct? Are the calculations accurate?

import unittest
import sys
import os
sys.path.insert(0, os.path.join(os.path.dirname(__file__), '..', 'scripts'))

from validate_expenses import parse_date, parse_amount, detect_duplicates, check_required_columns


class TestParseDate(unittest.TestCase):
    def test_valid_yyyy_mm_dd(self):
        valid, result = parse_date("2026-07-07")
        self.assertTrue(valid)
        self.assertEqual(result, "2026-07-07")

    def test_valid_mm_dd_yyyy(self):
        valid, result = parse_date("07/07/2026")
        self.assertTrue(valid)
        self.assertEqual(result, "07/07/2026")

    def test_invalid_date(self):
        valid, result = parse_date("not-a-date")
        self.assertFalse(valid)

    def test_empty_string(self):
        valid, result = parse_date("")
        self.assertFalse(valid)


class TestParseAmount(unittest.TestCase):
    def test_valid_amount(self):
        valid, amount = parse_amount("123.45")
        self.assertTrue(valid)
        self.assertAlmostEqual(amount, 123.45)

    def test_dollar_sign(self):
        valid, amount = parse_amount("$1,234.56")
        self.assertTrue(valid)
        self.assertAlmostEqual(amount, 1234.56)

    def test_invalid_amount(self):
        valid, amount = parse_amount("abc")
        self.assertFalse(valid)


class TestDetectDuplicates(unittest.TestCase):
    def test_no_duplicates(self):
        rows = [
            {"date": "2026-07-01", "description": "Lunch", "amount": "25.00"},
            {"date": "2026-07-02", "description": "Dinner", "amount": "40.00"},

15 Debugging Guide

When a skill does not work as expected, systematic debugging saves time. Below are the ten most common issues and how to resolve each one.

Likely cause
The activation description is too vague or does not match the user's phrasing. The agent cannot determine that this skill is the right one.
Diagnostic step
Compare the user's exact phrasing with the description. Check whether other skills have overlapping descriptions that might match first.
Corrective action
Add the user's actual phrasing to the description. Remove or narrow overlapping descriptions in other skills.
Example
User says "run the Monday report" but the description only mentions "weekly report." Add "Monday report" as a trigger phrase.
Likely cause
Two or more skills have overlapping activation descriptions. The agent picks the one that matches first.
Diagnostic step
List all skill descriptions and identify which ones could match the user's request. Check the agent's selection logic.
Corrective action
Make each skill's description more specific. Add negative conditions ("Do not use for expense reports") where overlap exists.
Example
Both "Weekly Report" and "Monthly Report" skills trigger on "generate report." Add time-specific phrases to each.
Likely cause
The SKILL.md instructions describe what to do in general terms but do not specify exact steps, file names, or formats.
Diagnostic step
Ask: "Could a new employee follow these instructions without asking questions?" If not, they are too broad.
Corrective action
Add specific file names, column names, output formats, and example values. Replace vague verbs with concrete actions.
Example
"Process the data" becomes "Read input.csv, validate that columns date, amount, and category exist, and write valid rows to output/valid.csv."
Likely cause
The instructions do not specify defaults or the skill does not provide enough context for the agent to proceed autonomously.
Diagnostic step
Review the questions the agent asks. Determine whether the answers could be anticipated and documented in the skill.
Corrective action
Add default values, specify the expected input format, and document what to do when optional information is missing.
Example
Agent asks "What threshold should I use?" Add "Use $1,000 as the default threshold unless the user specifies otherwise."
Likely cause
The script expects a file at a specific path, but the working directory is different from what the script assumes. Or the file has a different name than expected.
Diagnostic step
Check the working directory when the script runs. Print the full path the script is trying to open. Verify the file exists at that exact path.
Corrective action
Use absolute paths or resolve relative paths from a known base directory. Add a file-existence check at the start of the script with a clear error message.
Example
Script assumes sales.csv is in the current directory, but the agent runs it from the skill folder. Use os.path.join with the working directory.
Likely cause
The input CSV has different column names, different column order, or extra columns that the script does not handle.
Diagnostic step
Print the actual headers from the CSV and compare them against what the script expects. Check for case differences, trailing spaces, or encoding issues.
Corrective action
Normalize column names in the script (lowercase, strip whitespace, replace spaces with underscores). Add a column-name validation step with a clear error message listing actual vs. expected columns.
Example
CSV has "Amount ($)" but script expects "amount." Normalize by stripping special characters and lowercasing.
Likely cause
Different date formats in the input (07/01/2026 vs. 2026-07-01 vs. July 1, 2026) and the script only handles one format.
Diagnostic step
Print the raw date values from the input. Identify which formats appear. Check which format the script attempts to parse.
Corrective action
Support multiple date formats by trying each one in sequence. Add the date format to the skill's documentation so users know what is accepted.
Example
parse_date tries YYYY-MM-DD first, then MM/DD/YYYY, then MM-DD-YYYY. Accept whichever matches.
Likely cause
The skill relies on AI interpretation for steps that should be deterministic, or the input data changes subtly between runs.
Diagnostic step
Run the skill twice with identical inputs. Compare outputs line by line. Identify which parts differ and why.
Corrective action
Move any step that produces different results each time from the AI to a script. Use scripts for all calculations, formatting, and data transformations.
Example
AI generates a summary table but uses different column ordering each time. Replace with a Python script that produces identical output.
Likely cause
The skill assumes the input data has a certain structure, range, or quality that does not match reality.
Diagnostic step
Examine the actual input data. Compare it against the assumptions documented in the skill. Identify every mismatch.
Corrective action
Validate assumptions at the start of the skill. If an assumption fails, report a clear error instead of producing wrong results.
Example
Skill assumes amounts are always positive, but the CSV contains refunds as negative numbers. Add a check for negative amounts and flag them as exceptions.
Likely cause
The skill sends an email, writes to a database, or calls an API, but does not check or report the result of the action.
Diagnostic step
Add logging before and after every external action. Check return codes, HTTP status codes, and error messages.
Corrective action
Wrap every external action in error handling. Log success and failure. Report failures to the user immediately instead of continuing silently.
Example
Email sending function returns a status code but the skill ignores it. Add: if status != 200, log the error and notify the user.

17 Deployment and Maintenance

Deploying a skill is not a one-time event. Skills need ongoing maintenance, versioning, and ownership to remain reliable over time.

Personal vs. Shared

Start with a personal skill for your own use. Once it is reliable and other people need it, promote it to a shared location with proper version control and documentation.

Naming Conventions

Use clear, descriptive names. Avoid abbreviations. The folder name should match the skill's purpose. Example: weekly-kpi-report, not wkrpt.

Version Control

Store all skill files in a Git repository. Every change gets a commit. Tags mark releases. Branches isolate experiments from production.

Changelogs

Maintain a CHANGELOG.md in each skill folder. Document what changed, when, and why. This helps users understand whether they need to update their workflows.

Ownership

Assign an owner to every shared skill. The owner reviews changes, handles issues, and decides when the skill is retired.

Review Frequency

Schedule regular reviews (quarterly is common). Check whether the skill is still used, still correct, and still follows current standards.

Sample Data

Keep sample data in the examples folder. Update it when the input format changes. Use it for demonstrations and testing.

Script Change Approval

Changes to scripts that run in production should be reviewed by at least one person who did not write the change. This catches bugs and safety issues.

Retirement

When a skill is no longer needed, document why it is being retired, move it to an archive folder, and remove it from active use. Do not delete it without a record.
